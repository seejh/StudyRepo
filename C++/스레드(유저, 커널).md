C++에서 'std::thread'를 사용해 스레드를 만들면, 생성한 각 스레드는 운영체제(OS)가 관리하는 커널 레벨 스레드에 직접 매핑됩니다. 다음은 프로세스가 단계별로 어떻게 작동하는지, 특히 많은 스레드를 다룰 때 커널이 어떻게 관리하는지 설명합니다:

1. 스레드 생성
— 사용자 공간: 'std::thread'를 사용해 스레드를 생성하면, C++ 표준 라이브러리는 운영체제의 스레딩 API와 인터페이스합니다(POSIX 호환 시스템의 'pthread_create' 또는 Windows의 'CreateThread'와 유사).
— 커널 공간: 운영체제가 새로운 커널 스레드를 생성합니다. 이 커널 스레드는 CPU 스케줄링의 기본 단위로, 'std::thread' 생성자에서 지정한 코드를 실행하는 역할을 합니다.

2. 스레드 매핑
— 1:1 모델: 리눅스와 윈도우를 포함한 대부분의 최신 운영체제에서 사용되는 스레딩 모델은 1:1 모델입니다. 즉, 각 사용자 공간 스레드('std::thread')는 하나의 커널 스레드에 매핑됩니다.
— 스레드 제어 블록(TCB): 운영체제는 각 스레드에 대해 스레드 제어 블록(TCB)이라는 데이터 구조를 유지합니다. TCB에는 스레드의 ID, 레지스터 상태, 스택 포인터, 스케줄링 우선순위와 같은 정보가 포함되어 있습니다.
— 프로세스 제어 블록(PCB): 각 프로세스는 프로세스 내 모든 스레드와 메모리와 같은 공유 자원에 대한 정보를 포함하는 프로세스 제어 블록(PCB)을 가지고 있습니다.

3. 스레드 스케줄링
— 커널 스케줄러: 커널 스케줄러는 어떤 스레드가 CPU 시간을 할당할지 결정하는 역할을 합니다. 이를 관리하기 위해 라운드로빈, 우선순위 기반 등 다양한 알고리즘을 사용합니다.
— 컨텍스트 스위칭: 스케줄러가 한 스레드에서 다른 스레드로 전환할 때 컨텍스트 스위칭을 수행합니다. 컨텍스트 스위치 동안 CPU는 현재 스레드의 상태(예: 레지스터 값, 프로그램 카운터)를 저장하고 다음에 실행할 스레드의 상태를 로드합니다.
— 선점: 스레드의 시간 조각이 만료되거나 우선순위가 높은 스레드가 준비되면 스케줄러는 현재 스레드를 다른 스레드로 대체할 수 있습니다. 이는 특히 스레드가 많은 시스템에서 흔히 발생합니다.

4. 다수의 스레드 처리
— 메모리 오버헤드: 각 스레드는 자체 스택(많은 시스템에서 보통 1MB)과 일부 TCB 오버헤드가 필요합니다. 많은 수의 스레드를 생성할 때는 상당한 메모리를 소모할 수 있습니다.
— 자원 제한: 운영체제는 일반적으로 프로세스가 생성할 수 있는 스레드 수에 제한을 둡니다(예: 리눅스의 'ulimit'). 이 제한을 초과하면 추가 스레드를 생성하려 할 때 오류가 발생합니다.
— 스레드 풀: 과도한 스레드를 생성하지 않고 많은 작업을 관리하기 위해 스레드 풀이 자주 사용됩니다. 스레드 풀은 더 적은 수의 스레드를 재사용하여 많은 작업을 처리하여 스레드 생성 및 폐기와 관련된 오버헤드를 줄입니다.

5. 스레드 관리 및 삭제
— 스레드 종료: 스레드가 실행을 마치면, 'std::thread::join()'을 사용하여 종료(함수 종료)하거나 명시적으로 조인될 수 있습니다. 운영체제는 그 스레드와 관련된 리소스를 정리합니다.
— 고아 스레드: 스레드가 종료되고 결합되지 않은 경우, "분리된" 또는 "고아 스레드"가 됩니다. 일부 시스템에서는 제대로 처리하지 않으면 자원 누수가 발생할 수 있습니다.

6. 커널 수준 관리
— 부하 분산: 운영체제는 작업 부하 균형을 맞추기 위해 CPU 코어 간 스레드를 마이그레이션할 수 있습니다. 이는 다른 코어가 유휴 상태일 때 단일 코어가 과부하되지 않도록 보장합니다.
— 교착 상태 및 우선순위 반전: 커널은 데드락(스레드가 자원을 무기한 기다리는 경우)과 우선순위 역전(낮은 우선순위 스레드가 높은 우선순위 스레드에 필요한 자원을 보유하는 경우)과 같은 잠재적 문제를 처리해야 합니다.

7. 확장성 문제
— 컨텍스트 전환 오버헤드: 스레드 수가 매우 많을 경우 컨텍스트 전환 오버헤드가 상당해질 수 있습니다. 커널은 자주 스레드 상태를 저장하고 복원해야 하므로 성능이 저하될 수 있습니다.
— 동기화 오버헤드: 스레드는 뮤텍스나 기타 동기화 원시 장치를 사용하여 공유 자원 접근을 동기화해야 하는 경우가 많습니다. 이러한 자원에 대한 높은 경쟁은 성능 병목 현상으로 이어질 수 있습니다.
— CPU 친화도: 캐시 성능을 향상시키기 위해 스레드를 특정 CPU에 고정할 수 있습니다(CPU 친화도). 이로 인해 캐시 미스가 줄어들지만 부하 분산의 유연성은 감소할 수 있습니다.

8. 스레드 정리 및 복구
— 스레드 종료: 스레드가 실행을 완료한 후, 운영체제는 스택 메모리와 TCB와 같은 자원을 다시 회수합니다. 스레드가 조인된 경우, 스레드 함수의 반환 값을 가져올 수 있습니다.
— 고아화된 스레드 회수: 스레드가 제대로 조인되지 않은 경우, 운영체제는 프로세스가 종료되거나 가비지 컬렉션 기법을 사용해 결국 스레드를 정리하지만, 이는 효율이 떨어집니다.

*요약*

- 직접 매핑: 각 'std::thread'는 운영체제가 관리하는 커널 스레드에 매핑됩니다.
- 자원 집약적: 개별 스택, TCB, 컨텍스트 스위칭이 필요해 많은 수의 스레드가 메모리와 CPU 자원을 많이 소비할 수 있습니다.
- 커널 스케줄링: 운영체제의 커널 스케줄러가 CPU 코어 전반에 걸쳐 이러한 스레드를 관리하고 스케줄링하며, 효율적인 실행을 보장하기 위해 다양한 전략을 사용합니다.
- 확장성 문제: 스레드 수가 증가할수록 컨텍스트 전환 오버헤드, 메모리 소모, 동기화 등의 문제가 더 두드러지게 나타나 성능에 영향을 미칠 수 있습니다.

동시 동기가 높은 애플리케이션의 경우, 스레드 풀 같은 대체 모델이나 Go의 고루틴 같은 다른 동시성 모델을 사용하는 것이 더 효율적일 수 있습니다.
