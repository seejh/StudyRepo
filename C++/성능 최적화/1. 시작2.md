# 게임 서버 로직 스레드 1개
게임 로직은 플레이어, 몬스터, 아이템, 스킬 등이 서로 얽혀 있어서 멀티 스레드로 안전하게 짜는 것이 매우 어렵다.

* 락 누락 시 버그(데이터 레이스)
* 락 과다 시 성능 저하
* 데드락 위험

### 로직 스레드 1개의 한계를 다른 방법으로 극복
1. 서버를 여러 프로세스로 나누기
   * 존1 서버, 존2 서버, 존3 서버 각각 독립 프로세스
   * 각 프로세스가 로직 스레드 1개씩 보유
2. 로직 틱 최적화
   * 1틱(보통 50~100ms) 안에 처리할 수 있는 양을 늘리는데 집중
3. 무거운 연산 분리
   * 경로탐색(A*), 물리 연산 등 독립적인 작업만 별도 스레드
   * 결과 로직 스레드로 전달

### 로직 틱
```
0ms 틱 시작 -> 로직 처리 -> 50ms 틱 종료
50ms 틱 시작 -> 로직 처리 -> 100ms 틱 종료

로직 처리가 50ms를 넘는다면
0ms 틱 시작 -> 로직 처리 70ms 걸림
50ms -> 다음 틱 와야 하는데 아직 이전 틱 처리 중 -> 틱 밀림 -> 게임 버벅임
```

### 게임에서 틱이 필요한 이유
게임 상태는 실시간으로 매 순간 처리하는 것이 아니라 이산적으로 50ms 마다 한 번씩 지금 상태를 계산.

### 1초에 10틱을 하는 이유
유저 경험(인간 반응속도)과 관련이 있다.
```
인간 평균 반응속도 200~250ms
네트워크 지연(RTT) 30~100ms(국내 기준)

50ms 틱일 때
클라 입력, 서버 전달(15~50) -> 로직 처리(~50) -> 서버 처리 후 클라 전달(15~50)

최악의 경우 : 50 + 50 + 50 = 150 (인간 반응속도 200~250 미만)
```

틱이 높을수록
* 처리해야 할 연산량 증가
* 네트워크 패킷 증가
* 서버 부하 증가

틱은 서버 부하와 게임 반응성의 트레이드 오프

```c++
void logicThread() {
  auto nextTick = chrono::steady_clock::now();

  while (running) {
    nextTick += chrono::milliseconds(50); // 다음 틱 시간

    processLogic(); // 로직 처리

    auto now = chrono::steady_clock::now();
    if (now > nextTick) {
      // 로직 처리가 50ms 초과, 틱 기준을 현재로 리셋
      // 틱 초과가 자주 발생하면 로직 최적화가 필요하다는 신호
      logWarning("틱 초과:");
      nextTick = now;
    } else {
      this_thread::sleep_until(nextTick); // 다음 틱까지 대기
    }
  }
}

sleep_for
로직 처리 10ms + sleep 50ms = 60ms 간격, 틱이 점점 밀림
sleep_until
로직 처리 10ms + sleep 40ms = 50ms 간격, 일정하게 유지
```
