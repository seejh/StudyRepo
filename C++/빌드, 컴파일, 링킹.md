### 빌드
소스 코드를 실행 파일로 만드는 과정<br/>
전처리 -> 컴파일 -> 어셈블 -> 링킹, 이 4과정을 거친다.<br/>
링킹 과정까지 끝나게 되면 시스템에 따라(윈도우, 리눅스) 실행 파일을 생성한다.
윈도우의 경우 .exe, 리눅스 계열의 경우 elf 형태의 파일을 생성하고 이 두 실행 파일의
포맷이 다르기 때문에 같은 코드를 빌드하더라도 서로 다른 시스템에서 사용할 수 없다.

* 전처리: 전처리기 매크로 처리
* 컴파일: 소스 파일들을 어셈블리 명령어로 변환
* 어셈블: 어셈블리 명령어를 실제 기계어로 변환(.obj)
* 링킹: 목적 코드들을 모아 실행 파일 생성

### 전처리 (Preprocessing)
소스 코드를 주석, 공백, 전처리 토큰으로 분리하는 단계

### 컴파일 (Compile)
전처리 토큰 -> 컴파일 토큰으로 변환<br/>
컴파일러가 컴파일 토큰을 해석해 해석 유닛(Translate Unit, TU) 생성.<br/>
해석 유닛은 각 소스 파일 별로 하나씩 존재.<br/>

이 TU에는 "유일 정의 규칙"이라는 것이 있다.<br/>
```
유일 정의 규칙 (One Definition Rule, ODR)

TU에 존재하는 모든 변수, 함수, 클래스 등의 정의는 유일해야 한다.
inline이 아닌 경우, 전체 프로그램에서 유일해야 한다.
```
쉽게 말하면... inline이면 각 TU 별로 존재해도 되나, inline이 아니라면 <br/>
전체에서 유일해야 한다.<br/>

```
inline

최적화 힌트
원래 inline 키워드는 이 키워드가 명시된 함수를 호출하는 곳의 문장을 이 함수의 내용으로 치환시켜도 된다는 의미였다.
이것의 의의는 함수를 호출하는 작업이 공짜가 아니라는 데 있다.(다만 인라인화 되면 실행 파일의 크기가 더 커질 수 있다.)
현재 c++ 컴파일러는 inline 키워드를 명시하지 않더라도 컴파일러가 판단하기에 처리를 하는 것이 성능면에서 낫다고
판단되면 인라인 처리하고 그 반대의 경우도 마찬가지이다.

그래서 현재의 inline은 아래와 같이 해석하는 것이 좋다.
```

# 수정 필요
여튼 문법과 ODR을 준수해서 어셈블리 코드가 생성되고 이 어셈블리가 생성한 파일을 재배치 가능 파일(Relocatable)이라고 해서
해당 obj 파일의 심볼(함수, 변수)들의 위치가 아직 확정되지 않은 상태이며 

### 링킹 (Linking)
목적 파일(.obj)들과 외부 라이브러리들을 합쳐서 심볼(함수, 변수)들의 최종 위치 결정.
