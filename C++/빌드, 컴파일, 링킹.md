# C++ 컴파일
컴파일 과정 : <br/>
전처리 -> 컴파일 -> 어셈블 -> 링킹 <br/>
<br/>
1. 전처리 (Preprocessing)
#include, #define 같은 전처리기 매크로들을 처리하는 단계<br/>
컴파일러에서 수행<br/>
2. 컴파일 (Compile)
소스 코드(.cpp)를 어셈블리 명령어로 변환하는 단계<br/>
컴파일러에서 수행<br/>
3. 어셈블 (Assemble)
어셈블리 코드를 실제 기계어로 이루어진 목적 코드(.obj)로 변환하는 단계<br/>
어셈블러에서 수행<br/>
4. 링킹 (Linking)
각각의 목적 코드를 모아서 하나의 실행 파일로 만들어주는 단계<br/>
링커에서 수행<br/>

<img width="719" height="231" alt="image" src="https://github.com/user-attachments/assets/47712cf6-f01a-48ff-a201-f92eed589a0b" /><br/>
보통 전처리, 컴파일, 어셈블 단계를 뭉뜽그려서 컴파일 단계 하나로 생각해도 무방하다.<br/>
빌드 = 컴파일 + 링킹<br/>
컴파일 = .cpp -> .o<br/>

링킹 과정이 끝나게 되면 사용하는 시스템에 따라 각기 다른 형태의 파일들을 생성한다. 윈도우의 경우, 
Portable Executable이라 불리는 PE 파일 형식의 파일(.exe 파일)을 생성하고 리눅스의 경우, 
Executable and Linkable Format(ELF) 형태의 실행 파일을 생성한다.
이 두 실행 파일의 포맷이 다르기 때문에 같은 코드를 컴파일 하더라도 서로 다른 시스템에서 사용할 수 없다.

### 상세 설명
전처리가 끝나면 소스 파일 별로 해석 유닛(Translate Unit, TU)을 생성한다. 

TU가 제대로 생성되기 위해서는 문법 오류가 없어야 한다. 그 중에서 잘 인지하지 못하는 규칙이 있다.
```
각 TU에 존재하는 모든 변수, 함수, 클래스, enum, 템플릿 등등의 정의(Definition)은 유일해야 하고
inline이 아닌 모든 함수, 변수들의 정의는 전체 프로그램에서 유일해야 한다.

유일 정의 규칙(One Definition Rule, ODR)
```

### 정의(Definition)와 선언(Declaration)
선언: TU에 새로운 이름을 도입하거나, 기존에 선언된 이름을 재선언

예
int f(); <- f라는 함수를 선언
int a; <- a라는 변수 정의, 정의는 선언을 포함하는 개념으로, 선언된 개체를 완전히 정의함을 뜻한다.

변수에서..
선언과 정의는 동일하다. 예외의 경우가 있다.
int a <- 정의
다만 extern 지정자가 들어간 선언의 경우, 명시적으로 초기화 되지 않는다면 선언이다.
extern int a; <- 선언, 정의x 
extern int b = 1; <- 정의

클래스 정의 내부에 inline이 아닌 static 멤버는 정의다.
struct S {
static int a; <- 선언
inline static int b; <- 정의
};

### 유일 정의 규칙(ODR, One Definition Rule)
```
각 TU에 존재하는 모든 변수, 함수, 클래스, enum, 템플릿 등등의 정의는 유일해야 하고
inline이 아닌 모든 함수, 변수들의 정의는 전체 프로그램에서 유일해야 한다.
```
각 TU에 정의는 유일해야 한다.
int f(); // f 선언
int f(); // f 선언
정의가 아니기에 OK
int f() {} // f 정의
int f() {} // f 정의
중복 정의, ODR 위반

inline이 아닌 정의는 모든 TU에서 정의가 유일해야 한다.
inline인 정의는 아니어도 되나, 사용하고자 하는 TU안에 정의되어 있어야 한다.
// TU1
int f(); // 선언
// TU2
int f() {} // 정의
전체 프로그램(모든 TU가 합쳐졌을 때)에서 정의는 딱 하나가 있어서 OK
// TU1
int f() {}
// TU2
int f() {}
합쳐졌을 때 ODR 위반

따라서 보통 다른 파일에서 사용하는 함수를 정의할 때, 헤더파일에 함수의 선언을 써놓고 소스 파일에 함수의 정의를 쓴다.

### inline?
초기에 inline 키워드는 컴파일러에게 "이 함수를 호출하는 문장을 그냥 이 함수의 내용으로 치환시켜도 된다"라는 의미였고
사용자가 인라인을 사용하는 이유는 함수를 호출하는 작업이 공짜가 아니기 때문이다. 그 대신 함수를 호출하는 모든 부분이
인라인화된 함수의 코드로 치환되기에 실행 파일의 크기가 더 커질 수 잇는 문제가 있다. <br/>

현재의 c++ 컴파일러는 굳이 inline이라고 명시하지 않아도 인라인 처리하는 것이 성능면에서 낫다고 판단되면 자동으로
인라인 처리를 하고 그 반대의 경우도 마찬가지다. 그래서 현재의 inline은 아래와 같이 해석하는 것이 옳다.
```
이 함수는 여러 개의 TU에 정의되어도 된다.
```

그러면 반대로 inline이 아닌 함수의 경우, 해당 TU에 정의될 필요는 없지만 전체에서 반드시 1개만 있어야 한다.
// a.h
inline int SomeFunc() {}
// b.cpp
int main() { SomeFunc(); }
inline 함수는 헤더파일에 정의해도 상관 없다.

// a.h
inline int SomeFunc();
// a.cpp
inline int SomeFunc() {}
// b.cpp
#include"a.h"
int main() { SomeFunc(); }
이 경우 b.cpp의 TU에 SomeFunc()의 정의가 없기 때문에 에러가 난다.

클래스 내부에 정의되어 있는 함수들은 자동으로 inline이다.
```
class A {
int Func() {} // inline
};
컴파일러에 의해 인라인으로 인식되고, 이 클래스 파일을 여러 파일에 include 하더라도 문제가 없다.

아래와 같이 선언만 있는 경우, inline으로 분류되지 않는다.
= 모든 TU 전체에 정의가 딱 하나 있어야 한다
= 헤더 파일이 아니라 cpp 파일에 정의한다.
class A {
int Func();
}

int A::Func() {}
```
