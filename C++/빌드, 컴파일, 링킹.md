# 빌드, 컴파일
<img width="719" height="231" alt="image" src="https://github.com/user-attachments/assets/47712cf6-f01a-48ff-a201-f92eed589a0b" /><br/>
소스 코드를 실행 가능한 프로그램으로 만들어내는 과정은 아래와 같다.<br/>
전처리 -> 컴파일 -> 어셈블 -> 링킹<br/>

보통 전처리, 컴파일, 어셈블 이 3과정을 컴파일 단계라고 뭉쳐서 부른다. 컴파일 단계에서는 소스 코드를 
cpu가 해석할 수 있는 기계어(바이너리)로 변경하는 과정이다.<br/>

컴파일 후 링킹을 하여 실행 가능한 프로그램으로 만들고, 소스 코드를 실행 가능한 프로그램으로 만드는
이 전 과정을 빌드라고 한다.<br/>

1. 전처리 (Preprocessing)
#include, #define 같은 전처리기 매크로들을 처리하는 단계<br/>
컴파일러에서 수행<br/>
2. 컴파일 (Compile)
소스 코드(.cpp)를 어셈블리 명령어로 변환하는 단계<br/>
컴파일러에서 수행<br/>
3. 어셈블 (Assemble)
어셈블리 코드를 실제 기계어로 이루어진 목적 코드(.obj)로 변환하는 단계<br/>
어셈블러에서 수행<br/>
4. 링킹 (Linking)
각각의 목적 코드를 모아서 하나의 실행 파일로 만들어주는 단계<br/>
링커에서 수행<br/>

<br/>
링킹 과정이 끝나게 되면 사용하는 시스템에 따라 각기 다른 형태의 파일들을 생성한다. 윈도우의 경우, 
Portable Executable이라 불리는 PE 파일 형식의 파일(.exe 파일)을 생성하고 리눅스의 경우, 
Executable and Linkable Format(ELF) 형태의 실행 파일을 생성한다.
이 두 실행 파일의 포맷이 다르기 때문에 같은 코드를 컴파일 하더라도 서로 다른 시스템에서 사용할 수 없다.

### 상세 설명
전처리가 끝나면 소스 파일 별로 해석 유닛(Translate Unit, TU)이라는 것을 생성한다.
TU에 대해 자세하게 알기 전에 알아야 하는 사전 지식부터 알아보자.

### 정의(Definition), 선언(Declaration)
선언은 새로운 이름을 도입하는 것이고, 정의는 선언을 포함하는 개념이며 완전히 정의하는 것을 말한다.<br/>
구체적 설명보다는 아래 예제를 통해 이해
```
int Func(); // 함수 선언
int Func() { return 1; } // 함수 정의

class A; // 클래스 선언
class A {}; // 클래스 정의

struct S {
static int a; // 선언
inline static int b; // 정의
}
int S::a; // 정의
```

### 유일 정의 규칙(ODR, One Definition Rule)
```
각 TU에서 변수, 함수, 클래스, enum, 템플릿 등의 정의는 유일해야 한다.
inline이 아니면 전체 프로그램에서 유일해야 한다.
```
한 방에 이해하자면<br/> 
inline이다. -> 각 TU 단위로만 유일하면 된다.<br/>
inline이 아니다 -> 각 TU 단위가 아니라 전체에서 유일해야 한다.<br/>

```
int Func() // a.cpp (TU1) 선언
int Func() {} // b.cpp (TU2) 정의
정의가 1개라서 OK

int Func() {} // a.cpp (TU1) 정의
int Func() {} // b.cpp (TU2) 정의
ODR 위반, 중복 정의

inline Func() {} // a.cpp (TU1) 정의
inline Func() {} // b.cpp (TU2) 정의
inline -> 각 TU에서만 유일하면 된다 -> OK
```
따라서 보통 다른 파일에서 사용하는 함수를 정의할 때, <br/>
헤더파일에 함수의 선언을 써놓고 소스 파일에 함수의 정의를 쓴다.

### inline?
#### (원래는...)최적화 힌트
초기에 inline 키워드는 컴파일러에게 "이 inline 함수를 호출하는 곳의 문장을 그냥 이 함수의 내용으로 치환시켜도 된다"라는 의미였고
이것에 의의는 함수를 호출하는 작업이 공짜가 아니라는데 있다. 다만 함수를 호출하는 모든 부분이 인라인 함수의 코드로 치환되기에 실행 파일의
크기가 더 커질 수 있다.<br/>

컴파일러가 자동으로 인라인 처리를 하고 그 반대의 경우도 만찬가지다. 그래서 현재의 inline은 아래와 같이 해석하는 것이 좋다.

링킹 과정에서 목적 코드들에 정의된 심볼들(함수, 객체)의 위치를 확정 시킨다.
심볼들의 위치를 정할 때 사용되는 것으로 저장 방식 지정자라고 하는 것이 있다.

저장 방식 지정자 (Storage class specifier)
auto와 register라는 지정자가 더 있었지만 c++11, 17 이후로 사라지고 아래 4개만 있다.
* static
* thread_local
* extern
* mutable

이 키워드들로 심볼들의 두 가지 정보를 지정할 수 있다.
"저장 기간"과 "링크 방식"이다.

저장 기간(Storage duration)
프로그램에서 모든 객체들의 경우 아래 넷 중 하나의 저장 기간을 가진다.

1. 자동 저장 기간(automatic)
   보통 {} 안에 정의되는 경우로 코드 블럭을 벗어나면 자동 소멸되며, 우리가 일반적으로 생각하는
   지역 변수들이 여기에 해당된다.  
3. static
   프로그램이 시작될 때 할당되고 프로그램이 끝날 때 소멸된다.
   static 객체는 프로그램에서 유일하게 존재한다.
5. thread

6. dynamic
