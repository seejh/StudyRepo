# 정적 라이브러리
컴파일 타임에 링킹되며 라이브러리 코드가 실행 파일에 포함되는 형태로 라이브러리 코드가 실행 파일에
포함되어 있기에 실행 파일만 배포하면 된다. <br/>

하지만 그렇기에 실행 파일의 크기가 커지게 되며 해당 라이브러리를 사용하는
프로그램이 시스템에 1개가 아닐 경우에 메모리 효율은 떨어지게 된다.

### 정적 라이브러리 생성
```c++
1. 프로젝트 생성
비주얼 스튜디오 -> 새 프로젝트 생성 -> 정적 라이브러리(MyMathLib)
// MyCalculator.h
class MyCalculator {
public:
  void Add(int value);
  void Sub(int value);
  void GetTotal() const;
private:
  int m_total;
};

// MyCalculator.cpp
void MyCalculator::Add(int value) { m_total += value; }
void MyCalculator::Sub(int value) { m_total -= value; }
void MyCalculator::GetTotal() const { return m_total; }

// pch.h
#ifndef PCH_H
#define PCH_H
#include<iostream>
#endif
```
위 내용 작성 후 빌드를 하면 .lib 파일이 생긴다.<br/>

### 정적 라이브러리 링크
본 프로젝트에서 정적 라이브러리를 링크하려면 아래 3단계를 거쳐야 한다.
```
1. 헤더 파일 경로
프로젝트 속성 -> c/c++ -> 일반 -> 추가 포함 디렉토리 -> 헤더 파일 경로

2. 라이브러리 경로
프로젝트 속성 -> 링커 -> 일반 -> 추가 라이브러리 디렉토리 -> .lib 파일이 있는 폴더

3. 사용할 라이브러리 명시
프로젝트 속성 -> 링커 -> 입력 -> 추가 종속성 -> .lib 파일
또는 #pragma comment(lib, ".lib 파일") 으로도 가능하다.
```



#### 정적 라이브러리 PCH
라이브러리 내부 소스들만 PCH의 이점을 누리고, 외부 프로젝트에는 아무런 영향을 주지 않는 것이 정상
메인 프로젝트에 pch.h를 강제하면 안된다.
배포용 헤더에는 pch 코드를 넣으면 안 된다.

# 동적 라이브러리
런타임(프로그램 실행 중)에 라이브러리를 불러오는 방식으로 묵시적 링킹, 명시적 링킹 2가지 방법이 있다.<br/>
동적 라이브러리의 코드 영역은 프로세스와 독립된 공간으로 공유되고 데이터 영역은 각 프로세스별로 독립적으로
생성된다.<br/>


## 암시적 링킹 (Implicit Linking)
프로그램이 시작되면 자동으로 로드<br/>
(=프로세스가 main에 진입하기 전에 로드되고 프로세스가 종료되면 해제된다)<br/>
.lib 필요, 일반 함수처럼 사용 가능<br/>

### 암시적 링킹 동적 라이브러리 생성
프로젝트 생성 -> 동적 라이브러리 (MyDll) 생성
```c++
// MyMath.h
/*
dllexport, dllimport는 dll의 함수, 데이터, 클래스를 외부에 내보내거나(dllexport),  외부에서 가져오기 위해(dllimport) 사용
dllexport:dll을 만드는 쪽에서 사용, 기호(함수, 클래스)를 dll 외부에 노출
dllimport:dll을 사용하는 쪽에서 사용, 외부 dll에 정의된 기호를 가져오도록 컴파일러에게 알려줌


*/
#ifdef MYMATH_EXPORTS
#define MYMATH_API __declspec(dllexport)
#else
#define MYMATH_API __declspec(dllimport)
#endif

// c++ 스타일 이름 변형(네임 맹글링) 방지
extern "C" {
  MYMATH_API int Add(int a, int b);
}

// MyMath.cpp
// 헤더 파일에서 dllexport로 작동하게
#define MYMATH_EXPORTS
#include "MyMath.h"

int Add(int a, int b) {
  return a + b;
}
```
코드 작성 후 빌드하면 아래 2가지 파일(.lib, .dll)이 생긴다.<br/>
MyMath.dll(실제 구현부가 들어있음)<br/>
MyMath.lib(dll 안의 함수 위치 정보를 가짐)<br/>

### 암시적 링킹 동적 라이브러리 링크

```
#pragma comment(lib, "MyDll.lib")
#include"MyMath.h"

int main() { cout << Add(1, 1); }

```

GetProcAddress()로 함수 주소를 가져와 호출해야 한다.

### 명시적 링킹 동적 라이브러리 정리

```
명시적 연결 (Explicit Linking), 암시적 연결 (Implicit Linking)

명시적 연결
런타임에 필요한 순간에 직접 라이브러리를 로드하고 해제하는 방식

라이브러리 로드, LoadLibrary()
함수 주소 가져오기, GetProcAddress()
라이브러리 해제, FreeLibrary()

-----------------------------------

암시적 연결
.h, .lib, .dll
프로그램이 시작될 때 운영체제가 라이브러리를 자동으로 메모리에 올리는 방식

소스 코드 상단에 #include로 헤더 파일 추가, 프로젝트 설정에서 .lib 파일을 종속성으로 연결

프로그램 실행 파일과 같은 폴더나 시스템 경로에 .dll 파일이 반드시 존재해야 한다.
```

### 동적 라이브러리 추가 내용
#### Name Mangling
c++은 함수 오버로딩(동일한 이름의 함수를 매개변수만 다르게 하여 여러 개 정의하는 것)을 지원하기 위해
컴파일 시 함수의 이름을 내부적으로 변경하는 Name Mangling을 수행한다.<br/>

이로 인해 동적 라이브러리에서 함수를 내보낼(Export) 때 다른 언어나 명시적 링킹에서 함수 이름을 찾지
못하는 일이 생길 수 있다. 이를 방지하기 위해 c++에서 동적 라이브러리를 만들 때 보통 아래의 내용을 추가해서 C언어
스타일로 이름을 고정한다.
```c++
extern "C" __declspec(dllexport) int Add(int a, int b) { return a + b; }
```
